# Samsung Auto Trader

## 1. 프로젝트 개요

본 프로젝트는 한국투자증권 Open API의 모의투자 환경을 이용하여 삼성전자(`005930`)를 대상으로 자동매매 흐름을 구현한 Python 프로그램이다.

REST API만 사용하였으며, WebSocket은 사용하지 않았다. 또한 API 호출을 과도하게 반복하지 않도록 일정한 polling interval을 두고 자동매매 루프를 실행하도록 구성하였다.

본 프로그램은 실거래가 아닌 모의투자 환경에서 동작하는 학습용 자동매매 프로그램이다.

---

## 2. 거래 대상

본 프로젝트의 거래 대상은 삼성전자이다.

- 종목명: 삼성전자
- 종목코드: `005930`
- 시장: 국내 주식시장
- 거래 방식: 한국투자증권 Open API 모의투자 REST API

---

## 3. 주요 기능

본 프로그램은 다음 기능을 수행한다.

1. 환경변수 기반 API 인증 정보 관리
2. access token 발급 및 당일 token 재사용
3. 삼성전자 현재가 조회
4. 계좌 잔고 및 보유종목 조회
5. 삼성전자 보유 수량 확인
6. 최대 보유 수량 기준 추가매수 제한
7. 평균매입가 확인
8. 최근 종가 조회
9. 5일/20일 이동평균 계산
10. 이동평균선 기반 매수/매도 신호 판단
11. 평균매입가 대비 익절/손절 조건 판단
12. 조건 충족 시 지정가 매수/매도 주문 요청
13. 주문 후 잔고 재조회
14. 주문 전후 계좌 상태 비교
15. 거래 가능 시간에만 자동매매 루프 실행
16. 주요 실행 과정 로그 출력

## 6. 환경변수 설정

프로젝트 실행을 위해 프로젝트 루트 폴더에 `.env` 파일을 생성해야 한다.

`.env.example` 파일을 참고하여 다음 값을 입력한다.

```env
GH_ACCOUNT=12345678-01
GH_APPKEY=your_app_key
GH_APPSECRET=your_app_secret
KIS_BASE_URL=https://openapivts.koreainvestment.com:29443
```

각 환경변수의 의미는 다음과 같다.

| 변수명 | 의미 |
|---|---|
| `GH_ACCOUNT` | 한국투자증권 모의투자 계좌번호 |
| `GH_APPKEY` | Open API app key |
| `GH_APPSECRET` | Open API app secret |
| `KIS_BASE_URL` | 한국투자증권 모의투자 API base URL |

보안을 위해 실제 `.env` 파일은 GitHub에 업로드하지 않는다. `.gitignore`에 `.env`를 포함하여 API key, app secret, 계좌번호가 노출되지 않도록 처리하였다.

## 8. 거래 시간 기준

본 프로젝트는 삼성전자 국내주식을 대상으로 하므로 한국 주식시장 시간을 기준으로 자동매매를 수행한다.

```text
자동매매 시작 시간: 09:10
자동매매 종료 시간: 15:30
```

장 시작 직후의 급격한 변동을 피하기 위해 09:00이 아니라 09:10부터 자동매매가 시작되도록 설정하였다.

미국장 자동매매로 확장할 경우에는 거래 대상, API endpoint, transaction ID, 시장 시간 기준을 미국장에 맞게 변경해야 한다. 본 프로젝트는 삼성전자 국내주식 자동매매이므로 한국장 기준을 사용한다.

## 9. 기본 주문 방식

기본 자동매매 요구사항에 따라 현재가를 기준으로 지정가 주문을 사용한다.

기본 주문 가격은 다음과 같다.

```text
매수 주문 가격 = 현재가 - 1,000원
매도 주문 가격 = 현재가 + 1,000원
```

예를 들어 삼성전자 현재가가 322,500원이라면 다음과 같이 주문 가격을 계산한다.

```text
매수 주문 가격 = 321,500원
매도 주문 가격 = 323,500원
```

이 방식은 현재가보다 낮은 가격에 매수하고, 현재가보다 높은 가격에 매도하려는 지정가 매매 구조이다.

단, 지정가 주문은 주문 요청이 정상 접수되더라도 실제 시장 가격 조건이 맞아야 체결된다. 따라서 주문 요청 성공과 실제 체결 완료는 구분해서 해석해야 한다.

## 10. 수익률 개선을 위한 전략 로직

기본 요구사항에서는 현재가 기준 매수·매도 주문을 요구하지만, 실제 자동매매에서 조건 없이 반복 주문을 넣는 것은 위험할 수 있다.

따라서 본 프로젝트에서는 수익률 개선 가능성과 리스크 관리를 고려하여 다음 조건을 추가하였다.

### 10.1 이동평균선 전략

최근 종가를 기준으로 5일 이동평균과 20일 이동평균을 계산한다.

```text
5일 이동평균 > 20일 이동평균
→ 단기 상승 추세로 판단

5일 이동평균 < 20일 이동평균
→ 단기 약세 또는 하락 추세로 판단
```

### 10.2 보유 수량 및 추가매수 조건

본 프로젝트는 삼성전자 한 종목을 대상으로 하므로, 상승 추세가 유지될 경우 보유 중이더라도 추가매수를 허용하도록 구성하였다.

다만 자동매매 루프가 반복되면서 동일한 매수 신호에 의해 무한정 매수되는 것을 방지하기 위해 최대 보유 수량을 설정하였다.

```text
5일 이동평균 > 20일 이동평균
그리고 보유 수량 < 최대 보유 수량
→ 매수 또는 추가매수 가능

보유 수량 >= 최대 보유 수량
→ 추가매수 제한
```

본 프로젝트에서는 최대 보유 수량을 3주로 설정하였다.

### 10.3 익절 조건

보유 중인 삼성전자의 현재 수익률이 평균매입가 대비 3% 이상이면 익절 매도를 고려한다.

```text
현재 수익률 >= +3%
→ 익절 매도 조건 충족
```

### 10.4 손절 조건

보유 중인 삼성전자의 현재 수익률이 평균매입가 대비 -2% 이하이면 손절 매도를 고려한다.

```text
현재 수익률 <= -2%
→ 손절 매도 조건 충족
```

## 11. 최종 자동매매 로직

본 프로그램의 최종 자동매매 흐름은 다음과 같다.

```text
1. 현재가 조회
2. 최근 종가 조회
3. 5일/20일 이동평균 계산
4. 잔고 조회
5. 보유 수량 확인
6. 평균매입가 확인
7. 현재 수익률 계산
8. 매수/매도/관망 조건 판단
9. 조건 충족 시 주문 요청
10. 주문 후 잔고 재조회
11. 일정 시간 대기 후 반복
```

구체적인 주문 조건은 다음과 같다.

```text
보유 수량이 있고 현재가가 평균매입가보다 3% 이상 상승
→ 익절 매도

보유 수량이 있고 현재가가 평균매입가보다 2% 이상 하락
→ 손절 매도

보유 수량이 있고 5일 이동평균 < 20일 이동평균
→ 추세 약화로 판단하여 매도 고려

보유 수량이 최대 보유 수량보다 적고 5일 이동평균 > 20일 이동평균
→ 현재가 - 1,000원에 매수 또는 추가매수 주문

그 외
→ 관망
```

이 전략은 삼성전자 한 종목을 대상으로 하므로 상승 추세가 유지될 때 일정 한도 내에서 추가매수를 허용한다. 다만 반복 실행되는 자동매매 프로그램이 과도하게 매수하는 것을 막기 위해 최대 보유 수량을 설정하였다.

또한 보유 중인 종목의 수익률이 익절 또는 손절 기준에 도달하면 추가매수보다 매도 판단을 우선하도록 구성하였다.

## 12. 주문 후 잔고 확인

주문 요청 후에는 계좌 잔고와 보유 종목을 다시 조회한다.

이를 통해 주문 전후 계좌 상태가 달라졌는지 확인한다.

```text
주문 전 잔고 조회
→ 주문 요청 또는 관망
→ 주문 후 잔고 재조회
→ 주문 전후 보유종목 및 평가금액 비교
```

단, 지정가 주문은 주문 요청이 성공하더라도 가격 조건이 맞아야 실제 체결된다. 따라서 주문 요청 성공과 실제 체결 완료는 구분해서 해석해야 한다.

## 13. 토큰 재사용

`auth.py`에서는 access token을 발급받은 뒤 `token_cache.json`에 저장한다.

같은 날짜에 다시 프로그램을 실행하면 새 토큰을 발급받지 않고 기존 토큰을 재사용한다.

이를 통해 불필요한 인증 요청을 줄이고, API 호출을 더 안정적으로 관리할 수 있다.

```text
오늘 발급받은 token이 있음
→ token_cache.json에서 재사용

오늘 발급받은 token이 없음
→ 새 access token 발급
```

`token_cache.json`에는 인증 정보가 포함될 수 있으므로 GitHub에 업로드하지 않는다.

따라서 테스트 시간 제한을 임시로 해제하면 자동매매 루프 자체는 실행되지만, 실제 주문 접수 또는 체결 여부는 평일 정규 거래 시간에 추가 확인이 필요하다.

## 15. 주문 응답 해석

본 프로그램은 주문 요청 후 한국투자증권 모의투자 서버의 응답을 로그로 출력한다.

예를 들어 장외 또는 주말 테스트에서는 다음과 같은 응답이 발생할 수 있다.

```text
모의투자 주문이 불가한 계좌입니다.
```

이는 Python 코드 자체의 오류가 아니라, 한국투자증권 모의투자 서버가 해당 시점의 주문을 허용하지 않았다는 의미이다.

또한 주문 요청이 정상 접수되더라도 지정가 주문은 실제 가격 조건이 맞아야 체결된다. 따라서 본 프로젝트에서는 주문 요청 결과와 주문 후 잔고 조회 결과를 함께 확인하도록 구성하였다.

## 16. 보안 관리

API key, app secret, 계좌번호 등 민감한 정보는 `.env` 파일에 저장한다.

`.env` 파일과 `token_cache.json` 파일은 `.gitignore`에 등록하여 GitHub에 업로드되지 않도록 처리하였다.

GitHub에는 실제 인증 정보가 들어 있지 않은 `.env.example` 파일만 업로드한다.
